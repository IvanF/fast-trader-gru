TECHNICAL MODULE SPECIFICATIONS:

1. MARKET SCREENER (Go) - The Conductor
- Execute a loop every 60 minutes. Query Bybit V5 REST API for all USDT Perpetual tickers.
- Filter and extract symbols where `turnover_24h >= 50,000,000` (50M USDT).
- Publish the updated list of active symbols to a Redis channel `config:active_symbols`.

2. DATA INGESTION SERVICE (Go)
- Subscribe to Redis channel `config:active_symbols`. Dynamically open/close Bybit V5 WebSockets (orderbook.50.USDT and publicTrade.USDT) based on the current active list.
- For every WS event, aggregate data into lightweight JSON or MessagePack payloads.
- Publish raw updates directly to Redis using high-throughput Pub/Sub channels:
  - Orderbook data -> `market:orderbook:[SYMBOL]`
  - Trade stream data -> `market:trades:[SYMBOL]`

3. ML ENGINE & VECTOR MEMORY (Python 3.11)
- Async Core: Use `redis-py` with `asyncio` to listen to all active `market:*:*` channels.
- Feature Engineering: Maintain a rolling 300-second in-memory buffer per coin to compute: Order Book Imbalance, Cumulative Volume Delta (CVD), Order Flow Speed, and VWAP deviation.
- Inference (Late Fusion Models via ONNX Runtime on CUDA):
  - Pass orderbook metrics through 1D-CNN.
  - Pass order flow sequences through GRU.
  - Concatenate outputs into a single 'Master State Vector' ($V_{state}$).
- The Experience Engine (FAISS + Vector Matcher):
  - Query a RAM-cached FAISS Index FlatIP with $V_{state}$ to fetch the Top-10 closest historical market contexts.
  - Calculate the aggregated Win-Rate and historical PnL ($V_{memory}$) from these 10 matches.
- Decision MLP Layer: Combine $V_{state}$ and $V_{memory}$. Output: Order_Direction (LONG/SHORT/HOLD), Confidence_Score (0.0-1.0), and Volatility_Multiplier (float).
- If Confidence_Score > 0.85, publish the signal to Redis channel `orders:signals`.
- Continuous Learning: Listen to Redis channel `execution:results`. When a trade closes, add its original $V_{state}$ and actual PnL back into the FAISS index. Periodically persist the FAISS index to disk.

4. OMS / EXECUTION SERVICE (Go) - Adaptive Grid Maker
- Subscribe to Redis channel `orders:signals`.
- Upon receiving a trade signal for a symbol:
  a) Grid Construction: Calculate entry levels, Multi-level Take-Profit (TP), and Stop-Loss (SL). Scale all distances using the provided `Volatility_Multiplier` to dynamically adjust to current market noise.
  b) Liquidity Check: Fetch the latest orderbook snapshot from Redis. Identify the largest limit cluster (liquidity wall) within the range. Place the SL 2 ticks behind this wall for safety.
  c) Order Placement: Deploy Maker-only (limit) orders via Bybit V5 REST API.
  d) Time-Stop Guard: If an entry order is filled but fails to hit TP within a dynamic timeframe (N seconds), force-close the position via an aggressive limit chase or taker execution to eliminate decay risk.
- Upon position closure, publish the final trade layout (FAISS vector ID, execution details, net PnL) to Redis channel `execution:results`.
